# Orbital Chaos — Comprehensive Improvement Design

## Overview

Four-session improvement plan for orbitalchaos.online covering models, site UX, infrastructure, and research. Sessions are ordered by dependency: model work produces numbers that feed into site updates, which inform pipeline design, which finalizes the paper.

**Audience:** All equally — portfolio/hiring, research community, and site visitors.

## Decisions

| Question | Decision |
|----------|----------|
| SGP4 baseline | Both TLE-based and Keplerian |
| Live predictions | Pre-computed hourly via cron, served as static JSON |
| Scroll animations | Combined: counters on stats, stagger on cards, fade on text |
| Storm comparison | Interactive toggle on results table |
| Session structure | By category, dependency-ordered |
| Paper figures | Matplotlib plots + architecture diagram |
| Notification method | TBD (email, Slack, or log-only — decide in Session 3) |

## Session 1: Models & Data

**Goal:** Produce all numerical results that downstream sessions depend on.
**Requires:** RunPod GPU access, Space-Track.org account (free) for TLE data.

### 1a. SGP4 Baselines

**New file:** `scripts/eval_sgp4.py`

**Keplerian baseline:**
- Fit orbital elements from position data using least-squares optimization.
- Propagate with two-body mechanics (existing `src/models/baseline_sgp4.py` as starting point).
- Evaluate at 6h horizon on the same test set used for ML models.

**TLE-based SGP4:**
- Download TLEs from Space-Track.org for ISS, DSCOVR, MMS-1 over 2023-2025.
- Propagate using the `sgp4` Python library.
- Find closest TLE to each test window start, propagate 6 hours, compute MAE against actual position.

**Output:** `results/sgp4_baselines.json`
```json
{
  "keplerian": {"iss": 450, "dscovr": 8500, "mms1": 12000},
  "sgp4_tle": {"iss": 320, "dscovr": 9200, "mms1": 14000}
}
```

### 1b. Storm-Conditioned Evaluation

**New file:** `scripts/eval_storm_conditioned.py`

- Load solar wind data and align Kp timestamps with orbit test windows.
- Kp is 3-hour cadence; assign each test window the Kp value concurrent with its prediction start time.
- Split test set into: quiet (Kp ≤ 3), active (4 ≤ Kp ≤ 5), storm (Kp ≥ 6).
- Re-evaluate all 9 existing checkpoints on each subset.
- Also evaluate SGP4 baselines on same subsets.

**Output:** `results/storm_conditioned_mae.json`
```json
{
  "lstm": {
    "iss": {"quiet": 110, "active": 180, "storm": 350},
    "dscovr": {"quiet": 12500, "active": 13100, "storm": 14200},
    "mms1": {"quiet": 18200, "active": 19000, "storm": 20100}
  },
  "transformer": { ... },
  "multimodal": { ... },
  "sgp4_tle": { ... }
}
```

**Key hypothesis to validate:** Multi-modal ISS MAE during storms should be closer to (or better than) LSTM, because the solar wind branch provides drag correction signal that the orbit-only model lacks.

### 1c. Shorter Prediction Horizons

- Retrain LSTM and multi-modal on ISS only at 1h (60 steps) and 3h (180 steps) horizons.
- Same hyperparameters, same train/val/test split.
- 4 new checkpoints total.

**Output:** `results/horizon_comparison.json`
```json
{
  "lstm_iss": {"1h": 22, "3h": 68, "6h": 126},
  "multimodal_iss": {"1h": 25, "3h": 72, "6h": 175}
}
```

### 1d. Ensemble Model

- Average LSTM and multi-modal predictions on ISS test set (no retraining).
- Evaluate on all conditions (all, quiet, active, storm).

**Output:** Ensemble MAE added to existing results JSONs.

- `results/storm_conditioned_mae.json`

## Session 2: Site & UX

**Goal:** Make orbitalchaos.online visually polished and interactive with real data from Session 1.
**Requires:** Session 1 completed (results JSONs exist).

### 2a. Scroll Animations (Combined Style)

**CSS additions (~40 lines):**
- `.animate-in` base class: `opacity: 0; transform: translateY(20px); transition: 0.6s ease-out`.
- `.animate-in.visible`: `opacity: 1; transform: translateY(0)`.
- Stagger handled via JS-set `transition-delay`.

**JS additions (~50 lines):**
- Single `IntersectionObserver` with `threshold: 0.15`.
- On intersection: add `.visible` class, apply stagger delay for cards.
- Counter function for stat values: ease-out cubic from 0 to target over 1.5s.
- All elements observed once (unobserve after trigger).

**Applied to:**
- Stats grid (4.8M+, 3, 3 Years, 1 min): counter animation
- Spacecraft cards: staggered (120ms delay)
- Model cards: staggered
- Results table: fade
- Weather cards: staggered
- Pipeline steps: staggered

### 2b. Prediction Overlay on Globe

**Reads:** `public/predictions.json` (generated by Session 3 cron).

**Implementation:**
- On globe init, fetch `predictions.json`.
- If valid and <2h old, render predicted path as `arcsData` on the globe — a cyan dashed line extending from current ISS position 6 hours ahead.
- ~30 points along the predicted path (decimated from 360).
- Label "6h LSTM prediction" near the arc end.
- If JSON missing or stale, silently skip (no error shown to visitor).
- ~30 lines added to existing globe JS.

### 2c. Storm-Conditioned Toggle Table

**HTML changes:**
- Replace static `<table>` with same structure but wrapped in a container.
- Add 4 toggle buttons above: "All" (default), "Quiet (Kp≤3)", "Active (Kp 4-5)", "Storm (Kp≥6)".
- Add SGP4 row at bottom of table with gray "Physics" label color.

**JS changes (~40 lines):**

- Button click handler swaps table cell contents and updates `.best-value` highlighting.
- SVG chart updates to match selected condition.

**CSS changes (~10 lines):**
- `.toggle-btn` styling matching existing `.hf-link` style.
- `.toggle-btn.active` with cyan highlight.

### 2d. Mobile Polish

**CSS additions (~15 lines in `@media` block):**

## Session 3: Pipeline & Infrastructure

**Goal:** Automate predictions and monitoring so the site stays alive without manual intervention.
**Requires:** Sessions 1-2 completed, PyTorch CPU installed on server.

### 3a. Prediction Cron Job

**New file:** `scripts/cron_predict.py`

**Flow:**
1. Load LSTM ISS checkpoint (~660K params).
2. Fetch latest 24h of ISS positions from NASA SSC API (reuse `src/data/ssc_client.py`).
3. Preprocess (velocity derivation, normalize with saved stats).
4. Run inference (CPU, <5 seconds).
5. Denormalize output to physical coordinates.
6. Convert to lat/lon/alt via coordinate transform.
7. Decimate 360 points to ~30 for the globe.
8. Write `public/predictions.json`.

**Dependencies to install:**
- `pip install torch --index-url https://download.pytorch.org/whl/cpu` (~200MB, CPU-only wheels).

**Cron:** `0 * * * *` (every hour on the hour).

### 3b. Model Auto-Retrain

**New file:** `scripts/cron_retrain.py`

**Flow:**
1. Download latest data from HF dataset.
2. Append any new daily-fetched data from `data/` directory.
3. Retrain ISS LSTM with existing hyperparameters.
4. If validation loss improves over current best: push new checkpoint to HF.
5. If loss regresses or training fails: keep existing checkpoint, log warning.

**Cron:** `0 3 * * 0` (Sunday 3am UTC).
**Runtime:** ~10 minutes on 16 CPU cores.

### 3c. Health Monitoring

**New file:** `scripts/health_check.py`

**Checks:**
1. `curl -s orbitalchaos.online` returns 200.
2. ISS API (`wheretheiss.at`) responding.
3. NOAA API responding.
4. `predictions.json` exists and `generated_at` < 2 hours ago.
5. Disk space > 1GB free.
6. Data pipeline last ran < 36 hours ago (check `logs/`).

**On failure:**
- Write to `logs/health.log` with timestamp, check name, error detail.
- Notification method: TBD at implementation time (email/Slack/log-only).

**Cron:** `*/15 * * * *` (every 15 minutes).

### 3d. Cron Setup

Update `scripts/cron_setup.sh`:
```
# Existing
0 2 * * * /var/www/orbit/scripts/cron_fetch.py      # Daily data fetch
0 4 * * 0 /var/www/orbit/scripts/backup_s3.sh        # Weekly S3 backup

# New
0 * * * * /var/www/orbit/scripts/cron_predict.py     # Hourly predictions
0 3 * * 0 /var/www/orbit/scripts/cron_retrain.py     # Weekly retrain
*/15 * * * * /var/www/orbit/scripts/health_check.py  # Health monitoring
```

- `scripts/cron_retrain.py`
- `scripts/health_check.py`
- Updated `scripts/cron_setup.sh`

### 4b. Architecture Diagram

**File:** `paper/figures/architecture.png`

Matplotlib-drawn block diagram of the residual gated multi-modal architecture:

```
Orbit Input ──→ [Proj] ──→ [BiLSTM] ──→ [Base Head] ──→ base prediction ──┐
                              │                                             │
                              │ orbit sequence                              │  (+)  ──→ Output
                              ↓                                             │
                        [Cross-Attn] ←── [BiLSTM] ←── [Proj] ←── Solar Input│
                              │                                             │
                              ↓                                             │
                     [Attn-Weighted Sum]                                    │
                              │                                             │
                              ↓                                             │
                     [Perturbation MLP] ──→ perturbation ──→ (× gate) ──────┘
                                                               ↑
                              orbit hidden ──→ [Gate MLP] ──→ σ(gate)
```

Color-coded: cyan for orbit branch, pink for solar wind branch, green for gate. Boxes with rounded corners, arrows with labels.
